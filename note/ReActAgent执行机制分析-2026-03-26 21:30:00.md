# ReActAgent 执行机制分析

> 分析时间：2026-03-26 21:30:00

---

## 1、概述（Overview）

**分析对象：** `io.agentscope.core.ReActAgent`

**核心目标：** 理解 ReAct（Reasoning-Acting）循环的执行机制，包括迭代次数控制、reasoning/acting 阶段转换、工具执行流程。

---

## 2、原始问题（保真）

1. "ReActAgent 实现原理"
2. "`maxIters` 赋值逻辑是怎么样的"
3. "是执行一个tool，迭代次数加一下吗"
4. "如果迭代次数设置为5，那是一定会循环执行5次吗"
5. "那我如何判断一致执行是否符合预期，因为`isFinished`只是判断没有需要调用工具就返回"
6. "站在用户问题角度，是先执行acting还是`reasoning`，假设现在有一个问题需要调用三个工具，请梳理一下整个流程是如何执行的"
7. "如果LLM一次返回多个工具调用，详细分析一下acting并行执行3个工具并且3个结果都添加到记忆的流程"
8. "那这么说正常情况都是迭代次数>=`maxIters`后才会返回结果"

## 8、数据流

### 8.1 迭代次数变化

```
输入: maxIters = 5

iter=0: reasoning(0) → acting(0) → executeIteration(1)
iter=1: reasoning(1) → acting(1) → executeIteration(2)
iter=2: reasoning(2) → LLM 无工具 → 返回（提前结束）

实际迭代次数: 2（未达到 maxIters=5）
```

### 8.2 工具执行数据流

```
输入: 3 个 ToolUseBlock
    │
    ▼
List<Mono<ToolResultBlock>> = [
    Mono<tool1>,
    Mono<tool2>,
    Mono<tool3>
]
    │
    ▼
Flux.mergeSequential() → 并行执行
    │
    ├── Thread-1: tool1 → result1
    ├── Thread-2: tool2 → result2
    └── Thread-3: tool3 → result3
    │
    ▼
List<ToolResultBlock> = [result1, result2, result3]  // 保持顺序
    │
    ▼
串行处理每个结果 → memory.addMessage()
```

## 9、关键机制 / 设计点

### 9.1 迭代次数控制

| 要点 | 说明 |
|------|------|
| **默认值** | 10 |
| **增加时机** | acting 完成后 |
| **上限语义** | 不是固定次数，是最大次数 |
| **无验证** | 可设置 0 或负数（会导致异常） |

### 9.2 提前结束条件

| 条件 | 代码位置 | GenerateReason |
|------|----------|----------------|
| 无工具调用 | reasoning:476 | COMPLETED |
| HITL 停止 | reasoning:458 | REASONING_STOP_REQUESTED |
| 工具挂起 | acting:539 | TOOL_SUSPENDED |
| 达到上限 | reasoning:410 | MAX_ITERATIONS |

### 9.3 工具执行模式

| 模式 | 说明 |
|------|------|
| **并行执行** | 同一次 acting 中多个工具并行 |
| **结果串行处理** | 按顺序添加到记忆 |
| **顺序保持** | `mergeSequential` 保证结果顺序 |

## 10、关键结论

### 真实执行逻辑

1. **迭代次数**：完成一轮 reasoning + acting 后才 +1
2. **提前结束**：LLM 无工具调用即返回，不等待达到 maxIters
3. **并行执行**：同一次 acting 中多个工具并行执行
4. **maxIters 含义**：安全上限，防止无限循环

### 与直觉不一致的地方

| 直觉 | 实际 |
|------|------|
| 每执行一个工具 +1 | 完成一轮后才 +1 |
| maxIters=5 必定执行 5 次 | 最多 5 次，可提前结束 |
| 工具顺序执行 | 并行执行 |

## 11、错误理解记录

| 误解点 | ❌ 误解 | ✔ 正解 | 原因 |
|--------|--------|--------|------|
| 迭代次数增加 | 每执行一个工具 +1 | 完成一轮 reasoning+acting 后 +1 | 未仔细阅读 `acting()` 返回值 |
| maxIters 含义 | 固定执行次数 | 最大迭代上限，可提前结束 | 忽略了 `isFinished()` |
| 正常返回时机 | 达到 maxIters 才返回 | LLM 无工具调用即返回 | 忽略了推理结果判断逻辑 |

## 13、可复用经验

### 设计模式

| 模式 | 应用 |
|------|------|
| **Builder 模式** | ReActAgent.builder() 构建复杂对象 |
| **Hook 模式 (AOP)** | 扩展功能不修改核心流程 |
| **模板方法** | AgentBase.doCall() 固定流程 |

### 架构思想

1. **响应式编程**：所有操作返回 Mono/Flux
2. **事件驱动**：通过 HookEvent 拦截和修改流程
3. **关注点分离**：核心流程不依赖具体功能

### 代码技巧

```java
// 链式 Hook 调用
Mono<T> result = Mono.just(event);
for (Hook hook : hooks) {
    result = result.flatMap(hook::onEvent);
}

// 并行执行保持顺序
Flux.mergeSequential(monos).collectList()

// 判断执行结果
GenerateReason reason = msg.getGenerateReason();
```
